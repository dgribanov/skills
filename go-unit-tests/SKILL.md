# Go Unit Tests

## Instructions
Это инструкция по написанию коректных unit тестов в соответствии с лучшими практиками языка Go.
Назначение unit тестов — проверка отдельных функций и методов.

### Философия тестирования Go
В Go предпочитают простые, понятные тесты без магии. Никаких сложных фреймворков — только чистый код, который легко читать и понимать.

### Инструменты тестирования
- пакет testing из стандартной библиотеки Go
- пакет testify с дополнительными инструментами тестирования (assert, require, mock) - https://github.com/stretchr/testify
- пакет mockery для создания моков - https://github.com/vektra/mockery

### 🏗️ Анатомия теста в Go
Тесты в Go следуют простым, но строгим правилам:

Именование файлов: *_test.go — Go автоматически распознает тестовые файлы
Функции тестов: Начинаются с Test + нижнее подчёркивание и после него название тестируемой функции
Параметр теста: *testing.T для управления выполнением теста
Размещение: В той же папке, что и тестируемый код

Жизненный цикл теста:
Setup — подготовка данных для теста
Action — выполнение тестируемой функции
Assertion — проверка результатов
Cleanup — очистка ресурсов (если нужно)

Простейший пример теста:
```golang
// math.go - наша функция для тестирования
func Add(a, b int) int {
    return a + b
}
// math_test.go - тест функции
func Test_Add(t *testing.T) {
    result := Add(2, 3)
    if result != 5 {
        t.Errorf("Add(2, 3) = %d; ожидали 5", result)
    }
}
```

Что происходит в тесте:
- вызываем функцию с известными параметрами
- сравниваем результат с ожидаемым значением
- при несовпадении выводим подробное сообщение об ошибке

### 📊 Table-driven тесты
Табличные тесты — идиоматичный Go подход для тестирования множественных сценариев.

Почему table-driven тесты лучше?
- читаемость — легко увидеть все тестовые случаи
- поддерживаемость — добавление нового случая это одна строка
- DRY принцип — логика тестирования написана один раз
- отчетность — каждый случай получает отдельное имя в отчете

Структура table-driven теста:
```golang
func Test_Add(t *testing.T) {
    testCases := []struct {
        name     string
        a, b     int
        expected int
    }{
        {"положительные числа", 2, 3, 5},
        {"с нулем", 0, 5, 5},
        {"отрицательные", -2, -3, -5},
    }
    
    for _, tc := range testCases {
        t.Run(tc.name, func(t *testing.T) {
            result := Add(tc.a, tc.b)
            if result != tc.expected {
                t.Errorf("Add(%d, %d) = %d; want %d", 
                    tc.a, tc.b, result, tc.expected)
            }
        })
    }
}
```

Ключевые компоненты:
- testCases — срез структур с тестовыми данными
- t.Run() — создает подтест с именем для каждого случая
- анонимная функция выполняет actual проверку

- **`references/converter/converter_test.go`** — файл с unit тестом

### ⚠️ Тестирование ошибок
В Go ошибки это значения, и их тестирование критически важно для надежности.

Стратегии тестирования ошибок:
- positive path testing — проверка корректной работы
- error path testing — проверка правильной обработки ошибок
- edge cases — граничные случаи и неожиданные входные данные

Практический пример с ошибками:
```golang
func Divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, errors.New("деление на ноль")
    }
    return a / b, nil
}
func Test_Divide(t *testing.T) {
    // Тестируем успешный случай
    result, err := Divide(10, 2)
    if err != nil {
        t.Fatalf("Неожиданная ошибка: %v", err)
    }
    if result != 5 {
        t.Errorf("Divide(10, 2) = %f; want 5", result)
    }
    
    // Тестируем обработку ошибки
    _, err = Divide(10, 0)
    if err == nil {
        t.Error("Ожидали ошибку при делении на ноль")
    }
}
```

Важные моменты:
- всегда тестируйте и happy path, и error cases
- используйте t.Fatalf() для критических ошибок, которые - делают дальнейшее тестирование бессмысленным
- проверяйте не только наличие ошибки, но и её содержание

Пример табличного теста с тестированием ошибки находится здесь:
- **`references/version/version.go`** — файл с функцией, которую нужно протестировать
- **`references/version/version_test.go`** — файл с unit тестом

### Тестовые двойники
Для имитации внешних зависимостей тестируемой функции используются тестовые двойники. Они реализуются на основе интерфейса зависимости.

Тестовые двойники бывают трёх видов:
- stub - простая заглушка, реализующая методы интерфейса с определением возвращающаемых значений
- mock - делает то же, что и stub, а также сравнивает ожидаемые параметры вызова с фактическими и проверяет количество и последовательность вызовов методов интерфейса
- fake - полноценно эмулирует зависимость, упрощая её внутреннюю логику, возвращаемое значение зависит от параметров вызова

#### Stub
Это самый простой вид тестового двойника, предназначенный для возвращения заранее определённых значений. Он не следит за тем, как его вызывают, не записывает параметры и вообще не пытается имитировать логику настоящей зависимости. Если цель — проверить, как система реагирует на конкретный ответ.

Плюсы и минусы:
+ не хранит информацию о вызовах, что упрощает реализацию
- не позволяет проверять корректность вызова
- всегда возвращает ожидаемое значение, вне зависимости от параметров вызова

Когда применять?
- не требуется проверять корректность вызова, потому что методы интерфейса вызываются в тестируемой функции при любых условиях, одинаковое количество раз и с одинаковыми параметрами
- не требуется управлять возвращаемыми значениями методов интерфейса - всегда возвращаем одно определённое значение на каждый test case

#### Mock
Его цель — проверка корректности вызова методов интерфейса. Он следит за тем, как его вызывают, записывает параметры вызова и проверяет количество и последовательность вызовов методов интерфейса. Если вызываются методы интерфейса в неправильном порядке или с неправильными параметрами, то mock выдаст ошибку.
При этом mock, как и stub, будет возвращать ожидаемое значение, вне зависимости от параметров вызова.

Плюсы и минусы:
+ проверяет корректность вызова
- хранит информацию о вызовах, что усложняет реализацию
- всегда возвращает ожидаемое значение, вне зависимости от параметров вызова

Когда применять?
- требуется проверить корректность вызова методов интерфейса, потому что методы интерфейса вызываются в тестируемой функции при определённых условиях и/или с разными параметрами в зависимости от условий тестируемой функции
- не требуется управлять возвращаемыми значениями методов интерфейса - всегда возвращаем одно определённое значение на каждый test case

#### Fake
Его цель — имитация реальной зависимости. Он позволяет имитировать логику настоящей зависимости, возвращая ожидаемые значения в зависимости от параметров вызова. Если вызываются методы интерфейса в неправильном порядке или с неправильными параметрами, то fake выдаст ошибку, как и mock. Идеален для тестов, где требуется более реалистичное поведение зависимости, чтобы проверить интеграцию различных частей системы в условиях, приближённых к настоящим.

Плюсы и минусы:
+ имитирует логику настоящей зависимости, что позволяет проверить тестируемую функцию в приближенной к реальности ситуации
- требует внутренней реализации логики зависимости (упрощённой), что усложняет реализацию и удорожает дальнейшую поддержку

Когда применять?
- требуется имитировать логику настоящей зависимости, когда результаты вызова тестового двойника должны зависить от параметров вызова

#### Выбор типа тестового двойника
- если нет необходимости в более продвинутом тестовом двойнике, лучше использовать stub, чаще всего этого достаточно
- если нужно проверить корректность вызова, лучше использовать mock
- в исключительных случаях, когда нужно протестировать сложную логику вызова тестовых двойников, можно использовать fake
